Important design clarification about quaternions
The local engine must ACCEPT raw quaternion actions exactly as the API does. However, because v1 is deterministic and geometry-only, define validity as follows:
- any non-zero quaternion input is normalized
- the resulting oriented cuboid geometry is computed from the quaternion
- placements are allowed if they are geometrically valid and satisfy support/contact rules
- to keep support logic tractable in v1, only placements whose resting/support face is effectively horizontal within tolerance are considered stable/valid for placement
- continuous yaw around the vertical axis is allowed
- arbitrary quaternion input is still accepted at the API boundary, but unstable “edge balancing” or unsupported tilted resting poses should be rejected in v1
- add a future hook so a later settling/stability backend can accept more general poses

This means:
- transport/interface = full quaternion
- v1 deterministic validity = gravity-compatible resting configurations only

What a quaternion means in this codebase
Document this in README and code comments:
- orientation uses `[w, x, y, z]`
- normalize before use
- convert to rotation matrix for geometry
- compute oriented box corners from center, dimensions, and quaternion
- use quaternion everywhere at the API boundary and in stored state, even if support logic restricts which poses are valid in v1

Truck and scoring
Use the official truck dimensions:
- depth (x): 2.0 m
- width (y): 2.6 m
- height (z): 2.75 m

Coordinate convention:
- origin at front-bottom-left
- x increases toward the back of the truck
- y increases left-to-right across width
- z increases upward

Density:
density = total volume of placed boxes / (max_x_reached * truck_width * truck_height)
If no placed boxes, density = 0.

Geometry engine requirements
This is the hard part. Implement real geometry, not a toy placeholder.

1. Oriented cuboids
- from box center, dimensions, quaternion -> compute 8 world-space corners
- compute face normals and face polygons as needed
- compute AABB for rendering/debug only
- true collision uses OBB logic, not AABB-only shortcuts

2. Bounds checking
- all corners must remain inside the truck volume (with configurable epsilon)

3. Collision detection
- implement OBB-vs-OBB intersection using the Separating Axis Theorem (SAT)
- reject placements intersecting any already placed box

4. Support / stability validity for deterministic v1
- floor support always valid if bottom contact is on z=0 plane within tolerance
- otherwise require sufficient support from the union of top faces directly below
- support threshold = 90% of the candidate box’s support footprint area
- implement this in a deterministic, geometry-only way

Recommended practical support rule for v1:
- candidate resting face must be effectively horizontal
- identify its support footprint polygon projected onto the horizontal plane
- identify supporting top surfaces from boxes whose top face lies at the same support height within tolerance and whose top face is effectively horizontal
- compute overlap area between candidate support footprint and union of supporting top polygons
- valid if overlap area / footprint area >= 0.90

This deliberately excludes tricky edge/corner balancing and true rigid-body settling.

5. Contact snapping helpers
Implement utility functions to:
- snap a previewed candidate to the floor
- snap a previewed candidate onto the top of a supporting box / support plane
- optionally snap x/y gently to nearby boundaries for usability

6. Feasibility helpers
Implement search helpers to:
- test if the current box has any valid placement

RL / future agent hooks
I want future RL support, but the canonical engine observation must stay raw and variable-length.

Implement these RL-oriented pieces now, without forcing full Gymnasium tensor compliance:

1. `RawEpisodeEnv`
Methods:
- `reset(seed=None) -> raw_obs`
- `step(action: dict | np.ndarray) -> (raw_obs, reward, terminated, truncated, info)`

Behavior:
- raw observation is the same variable-length dict shape as the local API
- reward is sparse terminal density only:
  - all intermediate steps: 0.0
  - terminal natural completion: final density
  - timeout / failure terminal cases: 0.0 unless otherwise justified
- `terminated` for natural episode completion
- `truncated` for timeout-driven endings

2. Invalid RL action handling
Per my requirement, invalid RL actions should trigger an attempt to try another valid action instead of just ending or committing invalid actions.

Implement an `ActionRepairPolicy`:
- input: current raw observation + proposed raw action
- normalize quaternion
- try exact action
- if invalid:
  - search nearby translations
  - try snap-to-floor or snap-to-support-plane
  - optionally adjust yaw slightly
  - optionally project to nearest valid support plane
- use configurable attempt budget / time budget
- if a valid repaired action is found, commit it
- if none found before budget expires, handle as timeout/truncation or no-valid-action according to engine rules

This repair policy is for RL helper mode only.
The canonical local API should still reject invalid explicit placements with 422.
